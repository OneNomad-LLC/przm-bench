# Holdout Protocol

This document is the binding policy for the LongMemEval temporal-inference
holdout fixture. It is a social contract between everyone who works on or
with Engram — engineers, agents, reviewers, and anyone publishing benchmark
receipts.

## Files covered

| File | Role |
|------|------|
| `fixtures/longmemeval-temporal-inference-full-seen.json` | 80% training split — unrestricted |
| `fixtures/longmemeval-temporal-inference-full-holdout.json` | 20% holdout split — read-restricted |

## The core rule

**Holdout query texts SHALL NOT be read by any engineer or agent that makes,
reviews, or approves changes to Engram's retrieval pipeline, memory schema,
scoring logic, or any component that would be exercised during a benchmark
run.**

This includes:
- Reading the `query` field of any `queries` entry in the holdout fixture JSON
- Printing, logging, or rendering those queries in any tool, dashboard, or terminal
- Using the query texts — directly or paraphrased — as test cases during development
- Running the holdout fixture through any un-merged branch of Engram

The holdout is not a secret file. Its existence, its item count, and its ID
schema are public. What is restricted is reading its queries before publishing
a scored receipt against it.

## Who this applies to

Everyone. There are no exceptions for expediency, debugging, or curiosity.
If you have read the holdout queries, you are disqualified from making Engram
changes that could benefit from that knowledge until the next fixture rotation.

For automated agents (Claude Code, Cursor, etc.): the agent SHALL NOT read
holdout query texts as part of any task that involves modifying Engram. If an
agent needs to inspect a fixture for debugging purposes, it MUST use only the
seen fixture.

## How to publish a valid holdout score

1. Run the bench harness against `longmemeval-temporal-inference-full-seen.json`
   during development. Use this fixture for all tuning, debugging, and iteration.

2. When a release candidate is ready, run the bench harness against
   `longmemeval-temporal-inference-full-holdout.json` exactly once, on a clean
   tagged build, with `environment.git.dirty = false`.

3. Publish the holdout receipt alongside the seen receipt in the same PR or
   release commit. Never publish a holdout score without the corresponding
   seen score.

4. The published finding is always the pair: `(seen_R@10, holdout_R@10)`.
   Never surface only the holdout number.

## The delta rule

If `|seen_R@10 - holdout_R@10| > 3pp`, that gap IS the published finding.
It is not noise. It is evidence that the seen fixture was overfitted —
intentionally or not — and the holdout score is the honest number.

Do not discard, suppress, or footnote a delta exceeding 3pp. Publish both
numbers with the delta prominently labeled.

## Fixture rotation

The holdout is valid until one of the following events:

- A 3pp+ delta is published and investigated, and the team decides the fixture
  has served its purpose.
- The LongMemEval dataset is superseded by a materially different version.
- A new fixture version is generated with a fresh seed.

Rotation produces two new files with a bumped version suffix (e.g.
`-full-seen-v2.json` / `-full-holdout-v2.json`). The old files remain in the
repo as immutable historical artifacts.

## Reproducibility

The split was generated by `scripts/build-longmemeval-holdout.ts` using:

- Source: `xiaowu0162/longmemeval-cleaned` (MIT), `temporal-reasoning` subset, n=133
- Seed string: `MATT-2026-05-17`
- Seed derivation: SHA-256 of the seed string → first 4 bytes → big-endian uint32 →
  Mulberry32 PRNG → Fisher-Yates shuffle of the 133-entry index
- Split: first 80% (106 entries) → seen; remaining 20% (27 entries) → holdout
- Seed uint32: `0xba0b4e06` (3121303046)

Re-running the build script from the same source JSON produces byte-identical
fixtures. The seed is deterministic and publicly documented; the split is not a
secret. Only the query texts are read-restricted.

To regenerate:

```bash
# Ensure the dataset is present (check engram/benchmarks/data/ or download fresh)
export LME_DATA_PATH=/path/to/longmemeval_s_cleaned.json
npx tsx scripts/build-longmemeval-holdout.ts
npx tsx scripts/validate-fixtures.ts
```

## Design decisions

### A. Total fixture size: all 133

Using the full temporal-reasoning subset (n=133) rather than a fixed subsample
(e.g. n=100 or n=50) maximises statistical power per bench run. With R@10 near
99%, a 27-question holdout yields approximately ±2pp standard error — inside
the ±3pp detection threshold this protocol enforces. Subsampling to fewer
questions would widen variance without any benefit; using all available data is
the correct call.

### B. Items: full haystacks with distractors retained

Each LongMemEval question has its own isolated haystack of approximately 48
sessions. All sessions are included as `FixtureItem` entries — the 1-6 answer
sessions and all other (non-answer) sessions serve as distractors. This matches
the LongMemEval benchmark's intended difficulty: a retrieval system must find
the correct sessions inside a realistic ~48-session haystack, not against an
artificially small item set.

Item IDs are namespaced as `{questionId}:{sessionId}` to prevent collisions when
two questions share sessions from the same user's conversation history. Three
cases of the same raw session ID appearing twice within a single question's
haystack exist in the source dataset (dataset artifact); the build script
retains the first occurrence and drops the duplicate, logging a warning.
